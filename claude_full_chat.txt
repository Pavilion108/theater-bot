I cloned theater-bot, dug through the actual code (not just the README), and checked GitHub's issues/PRs/projects for it ÔÇö then built and tested the fix. Here's what's actually going on and what I did about it.

Root cause of the "hey" problem

Two real bugs, not one:

No intent router at all. The bot only understands /status, /restart, /cookies, STOP, and whatever bot_state currently expects. Say "hey" and it gets fed straight into _handle_location(), which tries to geocode "hey" as a place via Nominatim/Google Places/Overpass ÔÇö that's the "slow" you're feeling.
The bigger one: the whole bot is single-threaded. run()'s poll loop calls handle_message() inline, and that same call chain launches Selenium/Playwright browser automation. While a seat-selection step is running, Telegram simply isn't polled again ÔÇö so anything you send, including a greeting, waits behind whatever automation is mid-flight.
Root cause of "no Excel from photos/videos"

It's not a bug ÔÇö it's just not built yet. The poll loop only ever reads msg.get("text", ""); photo/video/document messages are silently dropped every time. There was zero media-handling code to fix.

What I built (implemented + tested end-to-end, not just designed)
Inline-keyboard menu on any greeting (hey, hi, /menu...) ÔÇö taps route to booking / media / chat / status, no typing needed.
Universal photo/video capture ÔÇö works in any bot state, always wins, exactly as you asked for "default" behavior.
Threaded dispatch with a state lock ÔÇö booking-flow messages still run one at a time in order (same guarantee as before), but they no longer block the poll loop. Greetings, media, and STOP now respond instantly even mid-automation.
media_intel.py ÔÇö extracts title/category/summary/OCR text/entities/key numbers from any photo or video.
excel_logger.py ÔÇö appends one row per photo/video to data/excel/media_log_<chat_id>.xlsx, auto-creating it on first use and growing forever after. I ran this end-to-end (fake image ÔåÆ extraction ÔåÆ workbook ÔåÆ verified rows accumulate).

Model choice: Gemini 2.0 Flash as the default ÔÇö free tier, and it's the only one of the two that natively handles video, not just photos, which fits the "public APIs, no extra cost" philosophy your movie_intel.py already follows. Claude is wired in as a drop-in alternative (EXTRACTION_BACKEND=claude in .env) for when OCR quality on messy/handwritten text matters more than cost ÔÇö its trade-off is it's paid and images-only.
